You have the **harness-workflow** plugin installed. It provides specification-driven development with OpenSpec integration, combining the OPSX workflow with Harness build-verify enforcement.

## The Rule

**Invoke relevant skills BEFORE any response or action.** Even a 1% chance a skill might apply means that you should invoke the skill to check. If an invoked skill turns out to be wrong for the situation, you don't need to use it.

## Prerequisites

Install OpenSpec CLI for full functionality:
```bash
npm install -g @fission-ai/openspec
```

## Available Commands

### Harness Workflow Commands

| Command | Purpose | When to Invoke |
|---------|---------|----------------|
| `/harness-workflow:init` | Initialize project with OpenSpec + Harness structure | New project, "set up harness", "init workflow" |
| `/harness-workflow:start` | Start a complete 5-stage development workflow | Feature request, "build me X", "start workflow", any creative work |
| `/harness-workflow:status` | Check current workflow state with content validation | "what stage am I at", "progress", "workflow status" |
| `/harness-workflow:verify` | Verify implementation against OpenSpec specifications | "verify", "check implementation", before committing |

### OPSX Commands (generated by `openspec init`)

These commands become available after running `/harness-workflow:init`:

**Core workflow (default):**

| Command | Purpose | When to Invoke |
|---------|---------|----------------|
| `/opsx:explore` | Think through ideas, investigate problems | "I want to explore", brainstorming, ideation |
| `/opsx:propose` | Create a change and generate all planning artifacts | "I want to propose a change", creating specs |
| `/opsx:apply` | Implement tasks from the change | "implement", "apply this change" |
| `/opsx:archive` | Archive when done | "archive", "done with this change" |

**Expanded workflow (via `openspec config profile`):**

| Command | Purpose | When to Invoke |
|---------|---------|----------------|
| `/opsx:new` | Start a new change scaffold | Starting a new change |
| `/opsx:continue` | Create next artifact incrementally | Building artifacts one by one |
| `/opsx:ff` | Fast-forward all planning artifacts | Quick path for all planning |
| `/opsx:verify` | Validate implementation against artifacts | Verifying change |
| `/opsx:sync` | Sync delta specs to main specs | Syncing specs |
| `/opsx:bulk-archive` | Archive multiple completed changes | Cleaning up |
| `/opsx:onboard` | Guided end-to-end walkthrough | Learning the workflow |

### OpenSpec CLI Commands

| Command | Purpose |
|---------|---------|
| `openspec init` | Initialize OpenSpec in project |
| `openspec new change <name>` | Create a new change directory |
| `openspec list --json` | List changes |
| `openspec status --json` | Show artifact completion status |
| `openspec validate --all --strict --json` | Validate all artifacts |
| `openspec show <item> --json` | Show change or spec details |
| `openspec instructions <artifact> --json` | Get enriched instructions |
| `openspec archive <change> -y` | Archive a completed change |
| `openspec schemas --json` | List available workflow schemas |
| `openspec templates --json` | Show resolved template paths |
| `openspec config profile` | Switch workflow profile |
| `openspec update` | Update instruction files |

## How Harness-Workflow Enhances OPSX

```
OPSX (Spec-driven development)       Harness-Workflow (Quality enforcement)
─────────────────────────────        ─────────────────────────────────────
/opsx:explore  → brainstorm          + superpowers:brainstorming integration
/opsx:propose  → create specs        + quality gate validation
/opsx:apply    → implement           + build-verify loop (lint/typecheck/test)
/opsx:archive  → archive             + entropy scan before commit
                                     + doom loop detection & recovery
                                     + 14-agent orchestration pool
                                     + git hooks / Claude Code hooks
```

```
# 1. Initialize (one-time)

# 2. Explore and plan

# 3. Or use the full automation

# 4. Check progress anytime

# 5. Verify before committing
```

## When to Use

**MANDATORY — invoke skill before any of these actions:**

- **Any feature/project request** → `/harness-workflow:start` (NOT optional)
- **Starting a new project** → `/harness-workflow:init`
- **Exploring an idea** → `/opsx:explore`
- **Creating specs for a change** → `/opsx:propose` or `/opsx:continue`
- **Implementing tasks** → `/opsx:apply`
- **Checking progress** → `/harness-workflow:status`
- **Final verification** → `/harness-workflow:verify`
- **Archiving a completed change** → `/opsx:archive`
- **User mentions "harness", "workflow", "openspec", "spec"** → invoke the relevant skill

## Quality Gates

The workflow enforces quality gates at each stage:
1. **Intent gate**: intent.md has Problem + Success Criteria
2. **Spec gate**: proposal, design, tasks have required sections (validated by `openspec validate`)
3. **Plan gate**: task breakdown with dependencies
4. **Execute gate**: build-verify passes, spec verification passes, entropy clean

## Integration with Other Plugins

harness-workflow works well with:

- **superpowers:verification-before-completion** — MUST be invoked before marking done (HARD-GATE)
- **superpowers:writing-plans** — For creating implementation plans from specs
- **OpenSpec CLI** (`npm install -g @fission-ai/openspec`) — Full OPSX command suite
