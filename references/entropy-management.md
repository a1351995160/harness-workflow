# Entropy Management

> Prevent "code rot" in AI-generated codebases through periodic cleanup, similar to a garbage collector

## Executable Script: entropy_scan.py

The skill provides `scripts/entropy_scan.py` for automated entropy detection:

```bash
python scripts/run.py entropy_scan.py                    # Run all scans
python scripts/run.py entropy_scan.py --json             # JSON output
python scripts/run.py entropy_scan.py --fix              # Auto-fix safe issues
python scripts/run.py entropy_scan.py --project-dir .    # Target directory
```

**Exit codes**: 0 = low entropy (<= 20 issues), 1 = high entropy (> 20 issues)

**Scan categories**:

| Scanner | Detects | Auto-fixable |
|---------|---------|-------------|
| `dead_code` | Unused exports, imports, variables | Yes (unused imports) |
| `style_drift` | Mixed quotes, semicolons, naming | Yes (quote style) |
| `stale_docs` | Docs referencing non-existent files/code | No |
| `todo_decay` | TODO/FIXME/HACK/XXX comments | No |

**JSON output structure**:
```json
{
  "project_dir": "/path/to/project",
  "language": "typescript",
  "total_issues": 12,
  "entropy_level": "LOW",
  "scanners": {
    "dead_code": {"count": 5, "issues": ["..."]},
    "style_drift": {"count": 2, "issues": ["..."]},
    "stale_docs": {"count": 1, "issues": ["..."]},
    "todo_decay": {"count": 4, "tags": {"TODO": 3, "FIXME": 1}}
  }
}
```

## The Problem

AI-generated codebases accumulate "AI Slop" over time, including:

| Entropy Type | Description | Harm |
|--------------|-------------|------|
| **Stale Documentation** | AI-generated comments/docs out of sync with code | Misleads subsequent development |
| **Circular Dependencies** | Implicit cycles introduced across multiple refactors | Slower builds, harder testing |
| **Dead Code** | Functions/modules generated by AI but never used | Increases maintenance burden |
| **Unused Imports** | Leftovers from Agent trying multiple approaches | Code noise |
| **Style Inconsistency** | Style differences across Agent sessions | Reduced readability |
| **Hardcoded Values** | Agent hardcodes values to quickly pass tests | Security and maintainability risks |
| **Bloat Boilerplate** | Agent tendency to generate excessive defensive code | Code bloat |

## Solution: Periodic GC Agent

Inspired by the OpenAI zero-human-code project — the team deployed **periodic background Codex tasks** to continuously scan the codebase, automatically detect deviations, and create fix PRs.


## Entropy Audit Checklist

### Documentation Consistency
- [ ] All public APIs are documented
- [ ] Example code in documentation is runnable
- [ ] Installation steps in README are still valid
- [ ] CLAUDE.md / AGENTS.md reflect current architecture

### Code Health
- [ ] No circular dependencies (detectable with `madge` or `dependency-cruiser`)
- [ ] No unused exports (detectable with `ts-prune` / `knip`)
- [ ] No duplicate code blocks (detectable with SonarQube)
- [ ] File sizes within limits (recommended < 400 lines)

### Security and Configuration
- [ ] No hardcoded secrets (Gitleaks scan)
- [ ] No hardcoded URLs or configuration values
- [ ] All environment variables are documented
- [ ] Dependency versions are up to date (no known CVEs)

### Style Consistency
- [ ] Consistent naming conventions (camelCase / snake_case)
- [ ] Consistent import style (relative/absolute paths)
- [ ] Consistent error handling patterns
- [ ] Consistent logging format

## Implementation

### Approach 1: Background GC Agent (Recommended)

```yaml
# .harness/config.yaml
entropy_management:
  enabled: true
  schedule: "0 9 * * 1-5"     # 9 AM every weekday

  gc_agent:
    model: sonnet              # Cost-effectiveness balance
    scope:
      - docs_audit             # Documentation consistency check
      - dead_code              # Dead code detection
      - dependency_check       # Dependency health check
      - style_consistency      # Style consistency check
      - hardcoded_values       # Hardcoded value detection

    actions:
      - create_refactoring_pr  # Create fix PRs
      - update_quality_grades  # Update quality grades
      - notify_team           # Notify team (optional)

    limits:
      max_prs_per_run: 5       # Max 5 PRs per run
      max_files_per_pr: 10     # Max 10 files per PR
```

### Approach 2: Integrated into Build-Verify Loop

```yaml
# Run entropy check after each build verification
build_verify_loop:
  post_verify:
    entropy_check:
      enabled: true
      threshold: "medium"      # low / medium / high
      auto_fix: false          # Report only, no auto-fix
```

### Approach 3: PR Review Integration

```yaml
# Check entropy during PR review
quality_gates:
  pre_merge:
    - name: entropy_scan
      command: "npm run entropy:check"
      blocking: false          # Warning only, non-blocking
      max_findings: 20
```

## GC Agent Scanning Tools

| Scan Item | Tool | Command |
|-----------|------|---------|
| Unused code | knip / ts-prune | `npx knip` |
| Dependency check | depcheck | `npx depcheck` |
| Circular dependencies | madge | `npx madge --circular src/` |
| Code duplication | jscpd | `npx jscpd src/` |
| Hardcoded secrets | gitleaks | `gitleaks detect` |
| File size | custom script | `find src -name "*.ts" -size +20k` |
| Outdated dependencies | npm outdated | `npm outdated` |

## Best Practices

1. **Prevention over cleanup** — Integrate entropy checks into the build-verify loop to catch issues at PR time
2. **Small batch fixes** — GC Agent processes only 5-10 files per run, avoiding large refactoring PRs
3. **Quality grade tracking** — Maintain quality grade trends and intervene proactively when degradation is detected
4. **Non-blocking** — Entropy reports default to non-blocking; serve as recommendations only
5. **Regular execution** — Run automated scans at least once daily
